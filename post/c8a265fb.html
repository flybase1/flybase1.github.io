SpringBoot+Security+JWT简单集成

1. Security

1.1 什么是Spring Security?

基于Spring AOP和Servlet过滤器的安全框架。
处理身份的验证以及授权。
减少大量的重复代码操作。
主要包含两个内容：
· 认证：确认用户可以访问当前网站
· 授权：用户拥有的功能权限
官网：https://spring.io/projects/spring-security/

1.2 为什么要用Security?

权限的设置就是为了让不同角色的用户访问到他们该访问的内容，没有使用Security之前，都是通过拦截器和监听器来控制用户的访问，很显然项目大了，用户角色多了，这样写后期维护比较麻烦，而且一旦增加了新的角色，就需要改动原本的监听器。而Security是Spring的一个安全模块，可以与Spring完美结合，进行少量的配置就能使用。

2. JWT

2.1 什么是JWT？

JSON Web Token (JWT)是一个开放标准(RFC 7519) ，它定义了一种紧凑和自包含的方式，用于作为 JSON 对象在各方之间安全地传输信息。此信息可以进行验证和信任，因为它是经过数字签名的。JWT 可以使用机密(使用 HMAC 算法)或使用 RSA 或 ECDSA 的公钥/私钥对进行签名。
虽然可以对 JWT 进行加密，以便在各方之间提供保密性，但是我们将关注已签名的Token。签名Token可以验证其中包含的声明的完整性，而加密Token可以向其他方隐藏这些声明。当使用公钥/私钥对对令牌进行签名时，该签名还证明只有持有私钥的一方才是对其进行签名的一方( 签名技术是保证传输的信息不被篡改,并不能保证信息传输的安全 )。

总结：JWT是目前最流行的跨域认证解决方案，是一种基于 Token 的认证授权机制。
官网：https://jwt.io/

2.2 JWT有哪些优点？

  1. 简洁(Compact)：可以通过URL，POST参数或者在HTTP header发送，数据量小，传输速度也很快；
  2. 自包含(Self-contained)：负载中包含了所有用户所需要的信息，避免了多次查询数据库；
  3. Token是以JSON加密的形式保存在客户端，所以JWT是跨语言的，原则上任何web形式都支持。
  4. 有效避免CSRF攻击：原本的使用Session存储，Session 认证中 Cookie 中的 session_id 是由浏览器发送到服务端的，借助这个特性，攻击者就可以通过让用户误点攻击链接，达到攻击效果。使用 JWT 的话，在我们登录成功获得 token 之后，一般会选择存放在 local storage 中。然后我们在前端通过某些方式会给每个发到后端的请求加上这个 token,这样就不会出现 CSRF 漏洞的问题。因为，即使有个你点击了非法链接发送了请求到服务端，这个非法请求是不会携带 token 的，所以这个请求将是非法的。但是这样会存在 XSS（Cross Site Scripting即跨站脚本）攻击中被盗的风险，为了避免 XSS 攻击，你可以选择将 token 存储在标记为httpOnly 的cookie 中。但是，这样又导致了你必须自己提供CSRF保护。

2.3 JWT的缺点？

  1. 服务器不保存Session的状态，而jwt是一次性的，修改jwt就意味着发一个新的jwt，意味着无法在中途废掉某个token。
    有两种解决方案，第一个就是将这个token放入redis里面存储，设置相应的过期时间，每次访问刷新token时间，但是这种方式就违反了JWt的这种无序的状态，jwt由无状态转变成了有状态；第二个是续签机制，改变JWT的有效时间，签发新的JWT，每次请求都刷新JWT，但是这样每次都要做jwt的加密解密，耗费性能。

2.5.1 Header

header里面是一个json对象

上面代码中，alg 属性表示签名的算法（algorithm），默认是 HMAC SHA256（写成 HS256）；typ 属性表示这个令牌（token）的类型（type），JWT 令牌统一写为 JWT。
最终这个 JSON 将由base64进行加密（该加密是可以对称解密的)

2.5.2 payload

也是一个json对象，存放实际传递的数据。JWT规定了7个官方字段

  1. iss (issuer)：签发人
  2. exp (expiration time)：过期时间
  3. sub (subject)：主题
  4. aud (audience)：受众
  5. nbf (Not Before)：生效时间
  6. iat (Issued At)：签发时间
  7. jti (JWT ID)：编号

也可以添加私有字段，JWT默认都是不加密的，一般不建议将秘密信息放在里面。这部分的声明也会通过base64进行加密,最终形成JWT的第二部分。

2.5.3 Signature

对前两部分进行签名，防止数据篡改。
这个签名的生成需要用到：

  • Header + Payload。
  • 存放在服务端的密钥(一定不要泄露出去)。
  • 签名算法。

2.6 如何基于JWT进行身份验证。

  1. 用户向服务器发送用户名、密码以及验证码用于登陆系统。
  2. 如果用户用户名、密码以及验证码校验正确的话，服务端会返回已经签名的 Token，也就是 JWT。
  3. 用户以后每次向后端发请求都在 Header 中带上这个 JWT 。
  4. 服务端检查 JWT 并从中获取用户相关信息。

2.7 如何防止jwt被修改?

服务端拿到jwt后会解析出header，payload，signature，服务端会根据header，payload再次生成一个signature进行比对，这里面尤其需要注意保管好密钥。
